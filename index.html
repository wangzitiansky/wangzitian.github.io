TCP协议

TCP首部格式

字段 作用
源端口号 目的端口号 多路复用或者分解来自上层或送到上层的应用数据
序号 用于实现可靠的数据传输服务
确认号 同上
首部字段长度 4bit 指示了以32bit的字为单位的TCP首部长度，通常20字节
选项字段 4bit 用于协商最大报文段长度
标志字段 6 bit 若为1 则对应的首部字段有效

流量控制

防止发送方发送报文段太快使得接收方缓冲区溢出，所有要有流量控制

TCP滑动窗口

TCPF发送方维护一个发送窗口，接受方维护一个接受窗口。接受方通过TCP报文中的窗口大小告诉发送方自己的窗口大小。

0窗口：

当接受窗口为0时，发送方不能发送

但是有两个例外：1. 紧急数据可以发送 2. 发送方可以发送1字节的TCP段，以便让接收方重新声明确认号和窗口大小，这种数据包叫窗口探测。

策略1：延迟确认

在没有可能进行捎带的情况下，接收方延迟发送确认段和窗口更新

策略2：Nagle算法

若数据每次以很少量的方式进入发送方，那么发送方就将第一个字符先发送出去，将后面到达的字符都缓存起来

当收到第一个字符的确认后，再将缓冲区的所有字符用一个TCP数据段发送出去，同时继续对到达的字符进行缓存

只有收到确认后，才继续发送下一个数据段

如果传递进来的数据能填充一般窗口或者一个最大数据段长度时候，允许发送一个新的数据段

不适用Nagle算法的情况

Internet网上玩游戏时，需要快速的短数据包流
可能与延迟确认相互作用，发生短暂死锁

策略3：Clark算法

Clark算法：

限制接收方只有在具备一般缓冲区为空，或者最大段长缓冲区为空时，才可以发送窗口更新

傻窗口症状：

接收端缓冲区已满，而接收方应用程序每次从缓冲区读取一个字节，那么每次只产生一个字节的窗口更新，，使得发送方只能发送一个字节

Nagle算法和Clark算法共同解决了傻窗口症状

拥塞控制

拥塞控制防止因为网络传输能力不足而导致的拥塞，超时，丢失等

发送方维护一个拥塞窗口cwnd

慢启动

最开始cwnd = 1MSS 并且每收到一个确认就加1 （指数增长）

如果出现了超时指示的丢包事件，则将ssthresh（慢启动阈值）设置为cwnd / 2并将cwnd设置为1并重新开始慢启动

当cwnd值超过ssthresh则进入拥塞避免模式

如果收到3个冗余ACK，则进入快速重传，并且快速恢复

拥塞避免

在此状态每个RTT只讲cwnd值增加一个MSS

就是收到所有ACK 才将cwnd加一

cwnd = cwnd + 1 / cwnd

当发生超时时候：

ssthresh = cwnd / 2 然后重新执行慢启动

快速恢复

当进入快速恢复时，令ssthresh = cwnd / 2 然后 cwnd = ssthresh

此时直接进入拥塞避免

网络对于丢包的反应，不像对于超时丢包的反应那么激烈！

从输入网址获得页面的过程

DNS解析域名

  1. 解析URI，生成HTTP请求消息，为了发送消息，我们需要知道域名对应的IP地址，所以下一步DNS

  2. 查询DNS，先浏览器缓存，之后操作系统缓存，之后本地Host文件，和向本地DNS服务器进行查询。

  3. 主机生成一个DNS报文，DNS被放入一个UDP报文段中，之后被放入一个IP数据包中，数据包的目的地之为DNS服务器的目的地址

  4. DNS被放入一个数据帧中，准备将该帧发给网关路由器

  5. 因为只知道网关路由器的IP地址而不知道MAC地址，所以要用到ARP协议

  6. 主机生成一个目的地址为网关路由器IP地址的ARP查询报文，并放入一个具有广播地址的数据帧中，子网内的所有主机都会受到该帧，包括网关路由器。

  7. 网关路由器得到该帧后不断向上分解得到ARP报文，发现其中目的IP地址和自己的接口的IP地址一致，则发送一个ARP响应报文，其中包含自己MAC地址，发送给主机

  8. 主机知道MAC地址后便可以继续DNS解析

  9. 发送DNS数据帧

  10. 网关路由器收到该帧后，根据其IP地址和自己的路由表项，决定下一跳地址

  11. 到达DNS服务器后，DNS服务器得到报文，并查询

  12. 若找到则返回一个相应报文

  13. 没找到则根据报文要求，递归或者迭代查询

HTTP请求
  1. 有了主机的IP地址之后，生成TCP套接字
  2. 先进行三次握手，生成一个TCP SYN报文
  3. 服务器收到之后，回应一个TCP SYN ACK报文
  4. 主机收到之后，回应一个TCP ACK报文，连接建立成功
  5. 浏览器生成HTTP报文，并委托套接字发送给服务器
  6. 服务器收到HTTP请求后，生成一个HTTP响应报文，放入web页面在报文主体中，发送给主机。
  7. 浏览器收到HTTP响应报文后，提取出内容，进行渲染

堆排序

堆是一个近似的完全二叉树 有如下性质
  • parent(i) = (i - 1) / 2
  • left(i) = 2 * i + 1
  • right(i) = 2 * i + 2
堆排序有三个关键步骤
步骤 作用 时间复杂度
heapify 维护堆的性质 O(lgn)
biuld_heap 建堆 O(n)
heap_sort 堆排序 O(nlgn)
  • heapify
heapify(arr, i, heap_size):
  l = left(i)
  r = right(i)
  if l <= heap_size and arr[l] > arr[i]:
      largest = l
  if r <= heap_size and arr[r] > arr[largest]:
      largest = r
  if largest != i:
      exchange arr[i], arr[largest]
      heapify(arr, largest, heap_size)
  • build_heap
build_heap(arr):
    heap_size = arr.length - 1
    for i = parent(heap_size) downto 0:
        heapify(arr, i, heap_size)
  • heap_sort
heap_sort(arr):
    build_heap(arr)
    n = arr.length
    for i = n - 1 to 1:
        exchange arr[0], arr[i]
        heapify(arr, 0, i)

网络层内部的工作方式

  • 数据报

    提供无连接服务，每个数据包携带完整的目的地址，独立进行路由选择

  • 虚电路

    提供面向连接的服务，发送数据包之前，必须先建立起一条从源路由器到目的路由器之间的路径，这个连接成为虚电路（vc）

    使用虚电路进行通信

    通信结束，释放虚电路

  • 两者比较

比较项目 数据报网络 虚电路网络
电路建立 不需要 需要
寻址 每个数据包包含全部源地址和目的地址 每个数据包包含一个短的虚电路号
状态信息 路由器不保留连接状态 针对每个连接，每条虚电路都需要保存其状态

ARP — 地址解析协议

为了解决什么问题？

因特网上主机通信使用的是IP地址，但是当一台主机发送数据包时，是根据硬件地址来确定目的接口的，链路层实现具体每段链路的通信，通信过程中MAC地址随着链路的改变而改变。ARP协议实现IP地址到MAC地址的映射。

如何实现

主机上有一个ARP高速缓存，里面有IP到MAC地址的映射表，若表中有则直接读出，若没有，则生成一个目的地址为需要查询的IP地址的ARP报文，该查询报文放入一个具有广播目的地址的以太帧中。 发送广播到以太网上，每台主机收到帧后，不断向上分解得到ARP报文，检查自己IP地址是否符合，若符合则发送一个ARP回答报文，其中包含它的MAC地址.

报文格式解析

以太网目的地址：全1为广播

以太网源地址

帧类型：对于ARP请求或者应答，该字段为0x0806

硬件类型：表示硬件地址类型，1表示以太网

协议类型：0x800表示IP地址

硬件地址长度，协议地址长度分别为6，4

操作字段指出四种操作类型，ARP请求，ARP应答，RARP请求，RARP应答

DHCP — 动态主机配置协议

介绍

允许服务器将IP地址和配置信息动态分发给客户端。正常情况下，DHCP服务器至少给客户端提供：IP地址，子网掩码，默认网关，也可提供DNS 服务器地址，

基本思想

每个网路必须有一个DHCP服务器来负责地址配置

当计算机启动时，它有一个嵌入在NIC中的内置以太网或其他链路层地址，但没有IP地址

该计算机在自己的网路上广播一个DHCP DISCOVER包。（若DHCP服务器和主机不在同一个LAN上，此时需要将路由器配置成一个DHCP中继代理）

当DHCP服务器收到请求，它就为该主机分配一个空闲的IP地址，并通过DHCP OFFER返回给主机

消息类型

DHCP 有 8 种消息类型，分别是 Discover、Request、Release、Inform、Decline、Offer、ACK、NAK。

其中前 5 种可由主机发起，后 3 种只能是 DHCP 服务端向主机发送，故 AB 正确。

ICMP — 网际控制报文协议

重要应用

ping： 主要用来测试两台主机之间的连通性。Ping 的原理是通过向目的主机发送 ICMP Echo 请求报文，目的主机收到之后会发送 Echo 回答报文。Ping 会根据时间和成功响应的次数估算出数据包往返时间以及丢包率。

NAT — 网络地址转换

为了解决IP地址短缺的问题
基本思想

为公司分配一个或少量共有IP地址，使用共有IP地址在Internet上传输数据包

公司内部，每台计算机有唯一的私有IP地址，使用该地址在公司内部传输数据包

当一个数据包离开公司网络时，需要地址转换

注：私有地址的数据包不应该出现在Internet上
工作过程

向外发送的数据包进入到NAT盒中，源地址被公有IP替代，传输层的源端口号被一个索引值替代

回来时根据传输层的目的端口号从NAT盒中找出原来的表项，提取相应的私有IP地址和原来的端口号，天道数据包中

缺陷

违反IP结构模型，IP结构模型声明每个IP地址均唯一标示一台机器

将Internet从一个无连接网络变成一个面向连接网络

违反最基本的协议分层规则

Internet上层的进程不一定适用TCP或者UDP

有些应用不一定适用TCP UDP协议

有些应用在正文插入了IP地址

TCP端口号16位 最多65536台机器被映射

class Reverse(object):
    """docstring for Reverse"""
    def __init__(self, data):
        self.data = data
        self.index = len(data)

    def __iter__(self):
        return self

    def __next__(self):
        if self.index == 0:
            raise StopIteration
        self.index -= 1
        return self.data[self.index]

# Generator 是一个用于创建迭代器的简单而强大的工具。 它们的写法类似标准的函数，但当它们要返回数据时会使用 yield 语句。 每次对生成器调用 next() 时，它会从上次离开位置恢复执行（它会记住上次执行语句时的所有数据值）。 显示如何非常容易地创建生成器的示例如下:
def reverse(data):
    for index in range(len(data)-1, -1, -1):
        yield data[index]
